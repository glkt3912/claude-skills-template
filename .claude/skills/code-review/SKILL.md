# 目的

コードの品質、可読性、保守性、パフォーマンスの観点からレビューを実施する。

# 手順

1. **コードの理解**
   - 変更の目的・背景を把握
   - 影響範囲を特定

2. **観点別レビュー**
   - **正確性**: ロジックは正しいか、エッジケースは考慮されているか
   - **可読性**: 命名は適切か、構造は理解しやすいか
   - **保守性**: 変更に強い設計か、重複はないか
   - **パフォーマンス**: 非効率な処理はないか
   - **セキュリティ**: 脆弱性はないか（詳細は security-audit スキル参照）

3. **フィードバック作成**
   - 問題点は具体的に指摘
   - 改善案を提示
   - 良い点も言及

# 出力フォーマット

```markdown
## レビュー結果

### 概要
[全体的な評価]

### 指摘事項

#### [Critical] ファイル名:行番号
- 問題: [具体的な問題]
- 理由: [なぜ問題か]
- 提案: [改善案]

#### [Warning] ファイル名:行番号
...

#### [Info] ファイル名:行番号
...

### 良い点
- [ポジティブなフィードバック]
```

# 重要度の定義

| レベル | 説明 |
| ------ | ---- |
| Critical | 必ず修正が必要（バグ、セキュリティ、重大な設計問題） |
| Warning | 修正を推奨（可読性、保守性の問題） |
| Info | 検討を提案（改善の余地、スタイル） |

# 制約

- 主観的な好みではなく、客観的な基準に基づく
- 指摘だけでなく改善案を提示
- 過度に細かい指摘は避ける
